Git et Github

Git

Git est un logiciel de gestion de version : il permet d'enregistrer l'état d'un projet (ensemble de fichiers) à un instant donné : on a ainsi la version xxx du projet. Il est toujours possible de revenir à une version antérieure si l'avancement du projet "a dérapé"...
Il est aussi possible de créer des branches divergentes de la branche principale (master), puis de fusionner les branches ultérieurement.

Il est nécessaire d'installer Git sur son ordinateur pour l'utiliser !

Après l'installation de Git, il faut juste une simple configuration en console :

  • git config --global user.name "votre identifiant"
  • git config --global user.email "votre adresse mail"

Github

Github est une plateforme qui permet d'héberger des projets (repositories) en ligne. Il faut se créer un compte pour y accéder. On peut évidemment consulter tous les projets publics de la communauté, et par exemple démarrer un projet personnel à partir d'un projet déjà existant (fork), ou même proposer des contributions (pull requests) sur des projets d'autres utilisateurs.

En couplant l'utilisation de Git et Github, on peut conserver l'état de l'avancement de ses projets en ligne en toute sécurité.

Créer et utiliser un "Repo"

Je me limiterai à une utilisation basique de Git et Github : un seul utilisateur qui travaille sur son projet dans son coin.

  • Étape 1 : créer un "Repo" sur Github
    • Se connecter à Github (sign in : id + mdp).
    • Créer un nouveau Repository (chercher le bouton New) : Donner une brève description du projet, conserver le mode Public et choisir Initialise with a README.
  • Étape 2 : cloner le "Repo" sur son ordinateur
    • Depuis Github, chercher le lien <adresse_du_repo> de clonage (bouton Clone or download) : l'adresse a le format https://github.com/identifiant/nom_du_repo.git
    • Sur l'ordi en console : git clone <adresse_du_repo>
  • Étape 3 : voir le résultat !
    • Se placer dans le dossier qui a été créé lors du clonage : le dossier porte le même nom que le Repository.
    • git status : donne l'état actuel du dossier vis-à-vis de Git.
    • Pour la suite des commandes, voir la section suivante !

Commandes Git principales

Encore une fois, je me contente de décrire les fonctionnalités de base :

  • Ajouter un fichier à l'index de Git, càd à l'ensemble des fichiers dont il doit "s'occuper".
    git add <nom du fichier> (pour ajouter un fichier spécifique à l'index de Git).
    git add . (pour ajouter tous les fichiers du dossier à l'index de Git).
  • Effectuer un Commit des dernières modifications : cela définit une nouvelle version du projet (à laquelle on pourra retourner un jour si nécessaire).
    git commit -m "message d'info du commit"
  • Envoyer les modifications sur Github.
    git push
    Remarque : Ici on "push" par défaut sur la branche master du Repo d'origine (utilisation basique, on a dit :)).
  • Récupérer l'état du projet depuis Github sur son ordi avant de faire des modifs.
    git pull
    Remarques importantes : Il est évidemment possible d'avoir une copie du projet hébergé sur Github sur plusieurs ordinateurs. Cela signifie qu'on aura fait un clonage du projet sur chaque ordi. Pour que l'ordi B soit à jour des modifs effectuées sur l'ordi A (et qui auront été "pushées" sur Github), il faut que l'ordi B récupère l'état en cours depuis Github : c'est le but de la commande git pull.
    Seulement après avoir effectué le pull, on pourra s'atteler à de nouvelles modifs avant de "pusher".
    Si on travaille vraiment seul, avec toujours le même ordinateur, ceci est donc inutile.

Je vais donc résumer les étapes de "travail quotidien" sur un projet. En parlant de "travail quotidien", je suppose qu'on travaille sur un ordinateur qui a déjà cloné un Repo, et pour lequel on veut "avancer" dans le projet.

  1. git pull
  2. modifs diverses
  3. git add .
  4. git commit -m "info sur les modifs"
  5. git push

Quelques commandes complémentaires de Git

Cette fois, je décris quelques commandes supplémentaires, mais en restant toutefois assez basique (ça peut donner des idées pour aller plus loin...) :

  • Connaître l'état de Git.
    git status
  • Connaître les branches existantes, et sur laquelle on travaille (voir l'astérisque).
    git branch
  • Créer une nouvelle branche.
    git branch <nouvelle_branche>
  • Changer de branche.
    git checkout <nouvelle_branche>
  • Voir la liste des commits git log
    Chaque commit est identifié par un identifiant SHA (plus auteur, date, commentaire).
  • Revenir à une version antérieure du projet.
    git checkout <SHA du commit désiré>

Git, Github et Visual Studio Code

L'éditeur de code que j'utilise et que je conseille (forcément puisqu'il me convient) est Visual Studio Code (VSC). Cet éditeur intègre une gestion de Git+Github.

Si on ouvre dans VSC un dossier qui a été initialement cloné depuis un Repo Github, on retrouve donc intégrées toutes les fonctionnalités décrites précédemment. Une barre de status apparaît en bas indiquant la branche active et une icône de synchronisation, et sur le côté gauche une icône de branche permet de gérer les commits.

  • Commencer par faire un Pull avec l'icône de synchronisation.
  • Faire toutes les modifs souhaitées
  • Pour faire un commit avec message, il faut cliquer l'icône dnas la barre gauche, entrer le message du commit et valider avec l'icône en forme de tick.
  • On finit enfin par un push avec l'icône de synchronisation en bas. S'ouvre alors (de façon pas très visible je trouve...) une zone de saisie pour l'identifiant et ensuite le mot de passe de connexion à Github.
